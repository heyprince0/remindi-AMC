I have a Next.js + Supabase + TypeScript app called Remindi (AMC management SaaS).
I recently added a team/multi-user feature that introduced an `org_id` column to
all business tables and changed RLS policies to scope data by organization instead
of individual user. The database migration is complete but the frontend code was
never updated. There are THREE separate bugs to fix.

═══════════════════════════════════════════════════════
BUG 1 — INSERTS ARE MISSING org_id
═══════════════════════════════════════════════════════

Every create/save operation is missing `org_id` in the insert payload.
RLS is currently DISABLED on affected tables as a temporary fix.

Tables that need fixing:
- customers
- contracts
- quotations
- invoices
- technicians
- service_history
- reminders_log

Search the entire codebase for these patterns:
  supabase.from("customers").insert
  supabase.from("contracts").insert
  supabase.from("quotations").insert
  supabase.from("invoices").insert
  supabase.from("technicians").insert
  supabase.from("service_history").insert
  supabase.from("reminders_log").insert

Also search for .upsert() calls on the same tables.

For EVERY occurrence found:

STEP 1 — Add this org_id lookup at the top of the submit/save handler,
before the insert call. Only add it once per function even if the function
has multiple inserts:

  const { data: membership } = await supabase
    .from("memberships")
    .select("org_id")
    .eq("user_id", user.id)
    .maybeSingle()

  const orgId = membership?.org_id
  if (!orgId) {
    toast.error("Organization not found. Please contact support.")
    return
  }

STEP 2 — Add org_id: orgId to the insert/upsert payload alongside the
existing fields. Do NOT remove user_id. Example:

  // BEFORE
  await supabase.from("customers").insert({
    user_id: user.id,
    name: formData.name,
    phone: formData.phone,
  })

  // AFTER
  await supabase.from("customers").insert({
    user_id: user.id,
    org_id: orgId,
    name: formData.name,
    phone: formData.phone,
  })

═══════════════════════════════════════════════════════
BUG 2 — SELECT QUERIES FILTER BY user_id, BLOCKING MEMBERS FROM SEEING DATA
═══════════════════════════════════════════════════════

Most fetch/load functions currently filter data like this:

  supabase.from("customers").select("*").eq("user_id", user.id)

This means an invited team member (who has a different user_id than the
admin who owns the data) sees zero results even though RLS correctly allows
them to read the org's data. The frontend query itself is blocking them.

Search the entire codebase for these patterns on all seven tables above:
  .eq("user_id", user.id)
  .eq('user_id', user.id)
  .eq("user_id", userId)

For EVERY occurrence found on SELECT queries (not insert/update/delete):
Remove the .eq("user_id", ...) filter entirely and let RLS handle the
scoping instead. Example:

  // BEFORE — member sees nothing because their user_id owns no rows
  const { data } = await supabase
    .from("customers")
    .select("*")
    .eq("user_id", user.id)
    .order("created_at", { ascending: false })

  // AFTER — RLS scopes to org automatically, member sees all org data
  const { data } = await supabase
    .from("customers")
    .select("*")
    .order("created_at", { ascending: false })

IMPORTANT: Only remove .eq("user_id", ...) from SELECT queries.
Keep it on INSERT and UPDATE calls — it still needs to be there for
row ownership tracking alongside org_id.
